# 유저 시나리오

Geas의 세 가지 실행 모드 -- Initiative, Sprint, Debate -- 에 대한 예측 사용 시나리오입니다. 세션 재개와 첫 온보딩 같은 변형 상황도 다룹니다.

---

## 시나리오 1: 새 제품을 밑바닥부터 만들기 (Initiative)

### 사용자

제품 아이디어는 있지만 코드베이스가 없는 개발자. 프로토타입이 아니라 아키텍처, 테스트, 문서, 보안 리뷰까지 갖춘 동작하는 MVP를 원합니다.

### 전제 조건

- `git init`이 완료된 빈 디렉토리
- 머릿속에 미션 아이디어가 있는 상태 (예: "스트릭과 대시보드가 있는 습관 추적 앱")
- Geas 플러그인 설치 완료

### 흐름

1. **사용자**가 미션을 자연어로 설명합니다.
2. **Intake**가 질문을 하나씩 던집니다 -- 범위 경계, 대상 사용자, 제약 조건, 성공 기준, 명시적 제외 사항. 트레이드오프가 담긴 접근 방식 2-3가지를 제시하고 방향을 고르게 합니다.
3. **사용자**가 각 질문에 답하고, 섹션별로 확인하며 확정합니다.
4. **Intake**가 명세를 `.geas/spec/seed.json`으로 확정합니다. 이 시점부터 seed는 변경 불가입니다.
5. **Nova**가 비전 선언문, MVP 범위, 사용자 가치 제안을 내놓습니다. PRD를 작성하고 사용자 스토리로 쪼갭니다.
6. **Forge**가 아키텍처와 기술 스택을 제안하고, 프로젝트 컨벤션을 `.geas/memory/_project/conventions.md`에 기록합니다.
7. **Circuit**, **Palette**, **Critic**이 아키텍처에 대해 병렬로 투표합니다. 전원 찬성이면 Discovery가 계속됩니다. 반대가 나오면 구조적 토론을 먼저 한 뒤 다시 투표합니다.
8. **Task Compiler**가 사용자 스토리를 인수 기준과 채점 rubric이 달린 TaskContract로 변환합니다.
9. MVP Build가 시작됩니다. 각 태스크마다 파이프라인이 실행됩니다: Implementation Contract → 구현 (worktree 격리) → 코드 리뷰 (**Forge**) → QA (**Sentinel**) → Evidence Gate → Critic Review → Nova Review → Retrospective → Resolve.
10. 모든 태스크가 통과하면 **Shield**가 보안 리뷰를 실행하고, **Scroll**이 문서를 작성합니다.
11. **Nova**가 최종 전략 요약을 내놓습니다. **Keeper**가 릴리스를 생성합니다 -- 버전 범프, 변경 로그, 최종 커밋.

### 결정 포인트

| 시점 | 결정 내용 |
|------|-----------|
| Intake | 미션 선언문, 범위 포함/제외, 인수 기준, 접근 방향 |
| 아키텍처 투표 | 제안된 아키텍처를 수락할지 (각 에이전트의 입장을 확인할 수 있습니다) |
| MCP 서버 설정 | 기술 스택에 맞는 MCP 서버를 연결할지 |
| Cut 판정 | Nova가 태스크를 Cut으로 판정하면, 동의하거나 재고를 요청합니다 |
| Iterate 피드백 | 태스크가 반복으로 돌아가면, 추가 가이드를 제공합니다 |

### 기대 결과

- 배포 가능한 상태의 동작하는 애플리케이션
- README, API 문서, 배포 가이드
- `.geas/spec/seed.json` -- 확정된 미션 명세
- `.geas/tasks/` -- 모든 기능의 TaskContract
- `.geas/evidence/` -- 코드 리뷰, QA 결과, Implementation Contract
- `.geas/decisions/` -- 아키텍처 투표 기록
- `.geas/memory/retro/` -- 태스크별 회고
- `conventions.md` -- 향후 Sprint를 위한 프로젝트 컨벤션
- 릴리스 태그 (예: `v0.1.0`)

### 대안 흐름

- **Evidence Gate 실패:** verify-fix 루프가 worktree에서 fixer 에이전트를 생성합니다. fixer가 실패를 수정하고 gate가 다시 실행됩니다. 재시도 예산은 3회입니다. 소진되면 **Forge**가 아키텍처를 리뷰하고, **Nova**가 전략적 판단을 내립니다 (Iterate 또는 Cut).
- **아키텍처 투표 불일치:** 반대한 에이전트가 자기 입장을 논증하는 구조적 토론이 실행됩니다. 토론 후 다시 투표합니다. 그래도 합의가 안 되면 **Nova**가 최종 결정을 내립니다.
- **Intake가 미션이 너무 넓다고 판단:** 분해안을 제시하고 첫 번째 서브프로젝트에 집중합니다. 나머지는 향후 작업으로 추적됩니다.
- **세션이 태스크 도중에 중단됨:** 시나리오 4를 참고하세요.

## 시나리오 2: 기존 프로젝트에 기능 추가 (Sprint)

### 사용자

기존 코드베이스에서 작업하는 개발자. 새 엔드포인트, UI 페이지, 서비스 통합 등 명확한 기능 하나를 추가하려 합니다. 전체 Initiative 수준의 검증을 태스크 하나에 적용하고 싶습니다.

### 전제 조건

- 아무 언어/프레임워크의 기존 코드베이스
- Git 저장소 초기화 완료
- 이전 Initiative 또는 Sprint에서 생성된 `.geas/` 디렉토리 존재 (없으면 시나리오 5 참고)
- `conventions.md` 작성 완료

### 흐름

1. **사용자**가 기능을 설명합니다. 하나의 한정된 추가. "주문 상태가 바뀌면 이메일 알림을 보내는 기능 추가."
2. **Intake**가 1-2개의 집중 질문을 합니다 -- 기능이 정확히 무엇을 하는지, 어떤 코드를 건드리는지, 무엇을 건드리면 안 되는지.
3. **Task Compiler**가 인수 기준과 rubric 차원이 담긴 TaskContract를 생성합니다. 기능 범위는 `seed.json`이 아니라 TaskContract에 들어갑니다.
4. 이 태스크 하나에 대해 전체 파이프라인이 실행됩니다: Design (**Palette**, UI가 있는 경우) → Tech Guide (**Forge**, 새 라이브러리나 모듈 간 변경이 있는 경우) → Implementation Contract → 구현 (worktree 격리) → 코드 리뷰 (**Forge**) → QA (**Sentinel**) → Evidence Gate → Critic Review → Nova Review → Retrospective → Resolve.
5. **Keeper**가 conventional commit 메시지로 커밋합니다.
6. 실행 요약이 감사 추적을 생성합니다.

### 결정 포인트

| 시점 | 결정 내용 |
|------|-----------|
| Intake | 기능 설명, 범위 경계, 건드리면 안 되는 것 |
| Implementation Contract | 코딩 전 제안된 작업 계획 검토 (선택 -- 기본적으로 에이전트가 승인) |
| Cut/Iterate | Evidence Gate나 Nova가 Cut 또는 Iterate를 권고할 때 |

### 기대 결과

- Evidence Gate를 통과한 기능이 main에 머지됨
- `.geas/tasks/{task-id}.json` -- 이 기능의 TaskContract
- `.geas/evidence/{task-id}/` -- Implementation Contract, 코드 리뷰, QA 결과, Critic 리뷰, Nova 판정
- `.geas/memory/retro/{task-id}.json` -- 회고
- Retrospective에서 새 패턴이 발견되면 `conventions.md` 업데이트

### 대안 흐름

- **Evidence Gate에서 `regression_safety` 실패:** 기존 테스트 스위트가 Tier 1 기계적 검증에서 실행됐는데 뭔가 깨졌습니다. fixer가 해당 회귀를 수정하고, 전체 테스트 스위트를 재실행한 뒤 gate가 재평가합니다.
- **기능이 너무 넓음:** Intake가 설명이 여러 독립 서브시스템을 다루고 있다고 지적하고, 별도 Sprint로 나누길 권장합니다.

## 시나리오 3: 코드 없이 의사결정 (Debate)

### 사용자

아키텍처, 기술 선택, 전략적 결정을 해야 하는 개발자 또는 엔지니어링 리드. 방향을 정하기 전에 여러 관점에서 구조화된 논증을 보고 싶습니다.

### 전제 조건

- 2-3가지 명확한 선택지가 있는 결정 사항
- 맥락 제공 가능 (현재 코드베이스 상태, 제약 조건, 왜 중요한지)
- 코드 작성은 불필요 -- 결정만 필요

### 흐름

1. **사용자**가 결정 사항을 설명하고 2-3가지 선택지를 제시합니다. "모놀리스를 마이크로서비스로 전환할지, 모듈러 모놀리스로 갈지, strangler-fig 패턴을 쓸지?"
2. **사용자**가 맥락을 제공합니다 -- 현재 문제점, 팀 구조, 제약 조건, 시간 압박.
3. **Forge**가 한 선택지를 기술적 근거와 함께 지지합니다.
4. **Critic**이 반대 논증을 펼치고 가정을 도전합니다.
5. **Circuit**이 백엔드/확장성 관점에서 평가합니다.
6. **Palette**가 UX/프론트엔드 관점에서 평가합니다.
7. 논증이 종합됩니다 -- 트레이드오프가 인정되고, 권고가 정리됩니다.
8. **사용자**가 최종 결정을 내립니다.
9. **DecisionRecord**가 `.geas/decisions/`에 기록됩니다.

### 결정 포인트

| 시점 | 결정 내용 |
|------|-----------|
| 프레이밍 | 질문, 선택지, 맥락 |
| 최종 결정 | 모든 관점을 들은 뒤 어떤 선택지를 택할지 |

### 기대 결과

- `.geas/decisions/`에 저장된 DecisionRecord:
  - 내린 결정과 그 이유
  - 검토된 모든 선택지의 장단점과 리스크
  - 트레이드오프 -- 얻는 것과 잃는 것
  - 어떤 에이전트가 어떤 입장을 취했는지
  - 각 에이전트가 인용한 구체적 기술 포인트

### 대안 흐름

- **토론 중 새 선택지 등장:** 에이전트가 원래 프레이밍에 없던 선택지를 제안합니다. 새 후보로 받아들이면 토론이 확장됩니다.
- **합의 불가:** 에이전트들이 근본적으로 의견이 갈립니다. 종합에서 불일치를 명확히 보여줍니다. 사용자가 각 에이전트의 입장 근거를 모두 확인한 뒤 결정합니다.

## 시나리오 4: 중단된 세션 재개

### 사용자

이전 세션이 태스크 도중에 중단된 개발자 -- 컨텍스트 윈도우 한도 도달, 터미널 종료, 머신 재시작 등.

### 전제 조건

- 진행 중인 태스크가 기록된 `.geas/state/run.json` 존재
- 체크포인트에 `current_task_id`, `pipeline_step`, `agent_in_flight`, `remaining_steps`가 기록되어 있음

### 흐름

1. **사용자**가 같은 프로젝트 디렉토리에서 새 세션을 시작합니다.
2. Geas가 진행 중 상태의 `run.json`을 감지합니다. 재개가 자동으로 시작됩니다.
3. Geas가 중단된 단계의 evidence가 `.geas/evidence/`에 이미 존재하는지 확인합니다.
4. **evidence가 있으면:** 세션 종료 전에 단계가 완료된 것입니다. `remaining_steps`의 다음 단계로 넘어갑니다.
5. **evidence가 없으면:** 단계가 완료되지 않은 것입니다. 처음부터 다시 실행합니다.
6. 파이프라인이 재개 지점부터 남은 단계를 거쳐 계속됩니다.

### 결정 포인트

없음 -- 재개는 자동입니다. 어디서 이어갈지 지정할 필요가 없습니다.

### 기대 결과

- 중단된 태스크가 남은 파이프라인을 모두 완료합니다
- 중복 작업 없음 -- evidence가 있는 단계는 건너뜁니다
- 세션이 중단 없이 계속된 것처럼 진행됩니다

### 대안 흐름

- **남은 태스크가 여러 개:** 중단된 태스크가 완료된 후, 의존성 순서에 따라 다음 태스크로 이동하여 정상적으로 계속됩니다.
- **evidence가 손상되었거나 불완전:** Geas가 해당 단계를 미완료로 간주하고 다시 실행합니다.

## 시나리오 5: 레거시 코드베이스에서 첫 Sprint

### 사용자

Geas를 한 번도 사용하지 않은 기존 코드베이스를 물려받거나 유지보수하는 개발자. 검증이 포함된 기능 추가를 원하지만, Geas가 프로젝트를 아직 모릅니다.

### 전제 조건

- 아무 언어/프레임워크의 기존 코드베이스
- Git 저장소 초기화 완료
- `.geas/` 디렉토리 없음
- 추가할 기능 있음

### 흐름

1. **사용자**가 기능을 설명합니다. "서비스 상태와 데이터베이스 연결을 반환하는 health-check 엔드포인트 추가."
2. Geas가 `.geas/` 디렉토리가 없음을 감지 -- 런타임 디렉토리를 초기화합니다.
3. **Forge**가 작업 시작 전에 코드베이스를 온보딩합니다:
   - 마커 파일(`package.json`, `go.mod`, `pyproject.toml` 등)을 스캔하여 스택 감지
   - 아키텍처 매핑 -- 진입점, 라우팅 패턴, 데이터베이스 연결, 핵심 모듈
   - 네이밍 컨벤션, import 스타일, 린터/포매터 설정 감지
   - 스캔 깊이 적응: 작은 프로젝트(~50파일)는 전체 스캔, 중간(50-500)은 집중 스캔, 큰(500+) 프로젝트는 대상 스캔
   - `.geas/memory/_project/conventions.md` 작성
4. **Intake**가 프로젝트 아이덴티티가 담긴 최소한의 `seed.json`을 생성합니다 -- 미션, 대상 사용자, 감지된 제약 조건. `"source": "sprint"`으로 표시하여 자동 생성임을 나타냅니다.
5. **Intake**가 기능에 대해 1-2개 질문을 합니다.
6. 전체 Sprint 파이프라인이 실행됩니다 -- 시나리오 2와 동일합니다.
7. **Scrum**이 회고를 작성하며, 발견된 새 패턴으로 `conventions.md`를 업데이트합니다.

### 결정 포인트

| 시점 | 결정 내용 |
|------|-----------|
| Intake | 기능 설명, 건드리면 안 되는 것 |
| 온보딩 검토 | `conventions.md`를 확인하여 Forge가 코드베이스를 올바르게 이해했는지 점검 (선택) |

### 기대 결과

- Evidence Gate를 통과한 기능이 main에 머지됨
- `.geas/` 디렉토리가 전체 런타임 상태로 초기화됨
- `conventions.md` -- Forge가 파악한 코드베이스 정보 (스택, 빌드 명령어, 주요 경로, 아키텍처 노트)
- `seed.json` -- 향후 Sprint를 위한 최소 프로젝트 아이덴티티
- 이후 Sprint에서는 온보딩을 건너뜀 -- `conventions.md`를 읽고 바로 시작

### 대안 흐름

- **Forge가 스택을 잘못 파악:** `conventions.md`에 부정확한 정보가 있습니다. `conventions.md`를 삭제하고 Geas에 재온보딩을 지시하면 다음 Sprint에서 처음부터 다시 스캔합니다.
