# Debate 모드 가이드

## 언제 사용하나요?

Debate는 **코드 없이 의사결정만 하는** 모드입니다. 에이전트들에게 이런 주제를 토론시키고 결론을 내리게 할 때 사용하세요:

- 아키텍처 선택
- 기술 선정
- 설계 방향
- 전략적 pivot
- 트레이드오프 분석
- 프로세스 결정

결과물은 **DecisionRecord** 하나입니다. 구현은 없고, 근거와 합의만 남깁니다.

## 동작 방식

**1. 질문 정하기**
- 어떤 결정을 내려야 하는지 설명하세요
- 선택지를 2-3개 정리하세요
- 맥락을 알려주세요 (기존 코드베이스, 제약 조건, 왜 이게 중요한지)

**2. 토론자 배정**
시스템이 각 입장을 맡을 에이전트를 띄웁니다:
- **Forge** -- 기술적 근거를 대며 한 옵션을 밀어줍니다
- **Critic** -- 반대 논거를 대고 가정에 태클을 겁니다
- **Circuit** -- 백엔드/확장성 관점에서 양쪽을 평가합니다
- **Palette** -- UX/프론트엔드 관점에서 양쪽을 평가합니다

**3. 정리**
각 논거를 요약합니다. 트레이드오프를 정리합니다. 추천사항을 남깁니다.

**4. 결정**
최종 결정은 사용자가 합니다 (또는 가장 강한 근거를 바탕으로 시스템이 결정합니다). **DecisionRecord**가 결정 내용, 근거, 참여자와 함께 `.geas/decisions/`에 기록됩니다.

## 결과물

각 debate는 아래 내용이 담긴 **DecisionRecord**를 남깁니다:

- **결정** -- 어떤 옵션을 골랐고 왜 골랐는지
- **검토한 옵션** -- 각각의 장점, 단점, 리스크
- **트레이드오프** -- 뭘 얻고 뭘 잃는지
- **참여자** -- 어떤 에이전트가 어떤 입장을 맡았는지
- **근거** -- 각 에이전트가 든 구체적인 기술적 포인트

## 팁

- **질문을 잘 잡으세요.** 질문이 명확할수록 토론의 질이 올라갑니다.
- **맥락을 충분히 주세요.** 기존 코드, 제약 조건, 시간 압박 같은 건 결과에 영향을 줍니다.
- **구조를 믿으세요.** Critic의 존재 이유는 허술한 논거에 태클을 거는 것입니다.
- **선택지를 좁히세요.** 2-3개가 적당합니다. 많아지면 노이즈만 늘어납니다.
- **결과를 활용하세요.** DecisionRecord는 나중에 "왜 이렇게 결정했지?"에 대한 추적 가능한 근거입니다.
