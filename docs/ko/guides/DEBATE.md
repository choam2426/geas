# Debate 모드 가이드

## 언제 쓰나

Debate는 **코드 없이 의사결정만 하는** 모드다. 에이전트들한테 이런 걸 토론시키고 결론을 내리게 할 때 쓴다:

- 아키텍처 선택
- 기술 선정
- 설계 방향
- 전략적 pivot
- 트레이드오프 분석
- 프로세스 결정

결과물은 **DecisionRecord** 하나다. 구현은 없고, 근거와 합의만 남긴다.

## 동작 방식

**1. 질문 잡기**
- 어떤 결정을 내려야 하는지 설명한다
- 선택지를 2-3개 정리한다
- 맥락을 준다 (기존 코드베이스, 제약 조건, 왜 이게 중요한지)

**2. 토론자 배정**
시스템이 각 입장을 맡을 에이전트를 띄운다:
- **Forge** -- 기술적 근거를 대며 한 옵션을 밀어준다
- **Critic** -- 반대 논거를 대고 가정에 태클을 건다
- **Circuit** -- 백엔드/확장성 관점에서 양쪽을 평가한다
- **Palette** -- UX/프론트엔드 관점에서 양쪽을 평가한다

**3. 정리**
각 논거를 요약한다. 트레이드오프를 정리한다. 추천사항을 남긴다.

**4. 결정**
최종 결정은 내가 한다 (아니면 가장 강한 근거를 바탕으로 시스템이 결정한다). **DecisionRecord**가 결정 내용, 근거, 참여자와 함께 `.geas/decisions/`에 기록된다.

## 출력물

각 debate는 아래 내용이 담긴 **DecisionRecord**를 남긴다:

- **결정** -- 어떤 옵션을 골랐고 왜 골랐는지
- **검토한 옵션** -- 각각의 장점, 단점, 리스크
- **트레이드오프** -- 뭘 얻고 뭘 잃는지
- **참여자** -- 어떤 에이전트가 어떤 입장을 맡았는지
- **근거** -- 각 에이전트가 든 구체적인 기술적 포인트

## 팁

- **질문을 잘 잡는다.** 질문이 명확할수록 토론의 질이 올라간다.
- **맥락을 충분히 준다.** 기존 코드, 제약 조건, 시간 압박 같은 건 결과에 영향을 준다.
- **구조를 믿는다.** Critic의 존재 이유는 허술한 논거에 태클을 거는 거다.
- **선택지를 좁힌다.** 2-3개가 적당하다. 많아지면 노이즈만 는다.
- **결과를 활용한다.** DecisionRecord는 나중에 "왜 이렇게 결정했지?"에 대한 추적 가능한 근거다.
