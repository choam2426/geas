# Reflection

## 목적

이 문서는 회고, memory 후보, debt 후보, gap, 후속 항목을 정의한다.

회고의 목적은 작업이 끝난 뒤 감상을 남기는 것이 아니라, 실행과 검증에서 드러난 사실을 다음 Mission의 계약, 실행 방식, 검증 방법, 기록 방식에 반영하는 것이다.

## 기본 관점

모든 회고 항목이 장기 기억이 되는 것은 아니다.

Geas는 작업 중 드러난 사실을 다음 범주로 나눈다.

- memory 후보
- debt 후보
- gap
- 후속 항목
- 버릴 항목

중요한 것은 많이 남기는 것이 아니라, 다음 작업에서 실제 판단을 바꿀 수 있는 항목을 남기는 것이다.

## Memory 후보

Memory 후보는 다음 Mission에서도 반복적으로 도움이 될 수 있는 교훈이다.

좋은 memory 후보는 다음 조건을 만족한다.

- 한 번의 우연이 아니라 반복 가능한 교훈이다.
- 다음 계약이나 검증 방법을 바꿀 수 있다.
- Evidence나 실제 실행 결과에서 나온다.
- 너무 특정 상황에만 묶이지 않는다.

예시는 다음과 같다.

- 특정 프로젝트에서는 문서 변경 전 항상 정의 문서를 먼저 읽어야 한다.
- 특정 검증 방법이 자주 빠지므로 계약에 명시해야 한다.
- 특정 역할이 자주 혼동되므로 roles 문서에서 경계를 확인해야 한다.

## Debt 후보

Debt 후보는 이번 Mission에서 해결하지 않았지만 나중에 비용이 될 수 있는 항목이다.

Debt 후보는 실패가 아니다. User가 현재 Mission의 목표와 범위 안에서 받아들인 남은 비용일 수 있다.

Debt 후보에는 다음이 들어갈 수 있다.

- 임시 해결
- 정리하지 못한 구조
- 나중에 분리해야 할 작업
- 검증을 더 강화해야 할 영역
- 운영이나 유지보수 비용을 만들 수 있는 선택

Debt 후보는 다음을 함께 남겨야 한다.

- 왜 이번 Mission에서 해결하지 않았는지
- 어떤 위험이 남는지
- 언제 다시 다뤄야 하는지

## Gap

Gap은 Mission 목표와 실제 결과 사이에 남은 차이다.

Gap은 단순한 후속 아이디어가 아니다. Mission의 definition of done, scope, acceptance criteria와 비교했을 때 드러난 차이다.

Gap에는 다음이 들어갈 수 있다.

- 목표 일부가 충족되지 않음
- 특정 환경에서 검증하지 못함
- User가 기대한 결과와 산출물이 다름
- Mission scope 안이라고 생각했지만 실제로는 처리되지 않음

Gap은 숨기지 않는다. User가 gap을 이해하고 수용할 수는 있지만, gap이 없는 것처럼 표현하지 않는다.

## 후속 항목

후속 항목은 현재 Mission 밖에서 다룰 수 있는 다음 작업 후보다.

후속 항목은 다음 중 하나로 분류한다.

- 새 Mission 후보
- 새 Task 후보
- 추가 검증 후보
- 문서 보완 후보
- 운영 또는 배포 확인 후보

후속 항목은 현재 Mission의 완료를 자동으로 막지 않는다. 다만 User가 수용 판단을 할 때 알아야 할 내용이면 반드시 드러내야 한다.

## 회고 절차

회고는 다음 순서로 진행한다.

1. Mission 결과와 User 수용 판단을 확인한다.
2. Evidence를 다시 본다.
3. self-check, review, verification에서 반복 가능한 신호를 찾는다.
4. memory 후보, debt 후보, gap, 후속 항목으로 분류한다.
5. 다음 Mission에 실제로 영향을 줄 항목만 남긴다.

## 회고 항목의 품질 기준

좋은 회고 항목은 다음 조건을 만족한다.

- 근거가 있다.
- 다음 행동을 바꿀 수 있다.
- 너무 크거나 추상적이지 않다.
- 특정 Mission의 맥락을 잃어도 이해할 수 있다.
- User가 받아들인 미검증 범위와 연결된다면 그 사실을 드러낸다.

나쁜 회고 항목은 다음과 같다.

- 단순 감상만 있다.
- 실행이나 Evidence와 연결되지 않는다.
- 다음에 무엇이 달라져야 하는지 알 수 없다.
- 모든 것을 장기 기억으로 남긴다.
- gap이나 debt를 성과처럼 포장한다.

## 수용 판단과 회고

회고는 User의 수용 판단 뒤에 온다.

완료로 수용된 Mission에서도 debt, gap, 후속 항목이 남을 수 있다. 반대로 중단된 Mission에서도 다음 작업에 유용한 memory 후보가 남을 수 있다.

중요한 것은 Mission을 좋게 보이게 만드는 것이 아니라, 다음 Mission을 더 정확하게 시작할 수 있게 하는 것이다.
