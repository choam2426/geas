# Geas 재정의 로드맵

## 1. 방향

이 작업의 중심은 v3.0.0 자체가 아니라 **Geas가 무엇인지 다시 정의하는 것**이다. v3.0.0은 그 새 정의를 처음 구현하는 버전이다.

기존 문서, 코드, 플러그인, 대시보드는 `.legacy/`에 보존하고 참고 자료로만 사용한다. v2.x의 개념, 이름, 상태, artifact는 필요가 확인될 때만 다시 가져온다.

로드맵의 목적은 세부 구현 계획을 쓰는 것이 아니다. 먼저 어떤 순서로 Geas를 다시 정의하고, 문서화하고, 구현할지를 정한다.

기본 방향은 다음과 같다.

- Geas의 정체성을 먼저 정의한다.
- 사용자 흐름과 작업 운영 흐름을 같은 문서층에서 다룬다.
- 문서 정본이 잠긴 뒤 구현을 시작한다.
- 구현 표면은 가볍게 시작한다.
- v3.0.0은 새 정의의 첫 구현 버전으로 다룬다.

## 2. 작업 흐름

Geas 재정의 작업은 아래 순서로 진행한다.

1. Geas 정의
2. 작업 운영 흐름 정의
3. 런타임 모델 정의
4. CLI 최소 표면 정의
5. Skills 흐름 정의
6. Agents 모델 정의
7. Plugin packaging 정의
8. 대시보드가 읽을 정보 정의
9. 구현 계획 작성
10. 구현
11. Dogfood
12. v3.0.0 릴리스 준비

각 단계는 앞 단계의 결정을 바탕으로 진행한다. 뒤 단계에서 앞 단계의 결정이 흔들리면, 구현으로 밀고 나가지 않고 해당 문서로 돌아가 수정한다.

## 3. 단계별 의미

### 1. Geas 정의

Geas가 무엇인지, 어떤 문제를 해결하려는지, 도달해야 할 목표와 핵심 원칙이 무엇인지 정의한다. 이 단계에서는 v2.x의 기능 목록이 아니라 Geas의 정체성과 사용자에게 제공하는 가치를 먼저 정한다.

### 2. 작업 운영 흐름 정의

사용자가 경험하는 흐름과 작업 운영 흐름을 함께 정의한다. 예를 들어 `intent -> contract -> work -> Evidence -> decision` 같은 중심축이 여기서 결정된다.

이 단계의 정본 문서군은 `docs/operating/`이다.

### 3. 런타임 모델 정의

핵심 흐름을 디스크에 어떤 artifact로 남길지 정한다. `.geas/` 구조, artifact 이름, 각 artifact가 책임지는 정보의 경계를 여기서 잡는다.

### 4. CLI 최소 표면 정의

런타임 모델을 쓰고 읽기 위한 CLI의 최소 명령 표면을 정한다. 이 단계에서는 CLI가 제품 판단을 소유하지 않도록 경계를 분명히 한다.

### 5. Skills 흐름 정의

사용자 entrypoint와 내부 sub-skill이 어떤 순서로 동작할지 정한다. 이 단계에서는 `/geas:mission` 같은 호출이 어떤 질문을 하고, 어떤 문서를 참고하고, 언제 CLI/runtime과 연결되는지 정의한다.

### 6. Agents 모델 정의

Geas에 어떤 agent 역할이 필요한지 정한다. v2.x의 큰 roster를 자동 계승하지 않고, 기본 흐름에 꼭 필요한 역할과 고위험 작업에서만 추가할 역할을 나눈다.

### 7. Plugin packaging 정의

Codex/Claude 같은 agent client에 skills, agents, CLI, assets를 어떤 구조로 노출할지 정한다. 이 단계는 사용자 대화 흐름을 다시 설계하는 단계가 아니라, 앞에서 정의한 흐름을 plugin package로 묶는 단계다.

### 8. 대시보드가 읽을 정보 정의

대시보드는 우선 읽기 표면으로 본다. 어떤 runtime artifact를 어떤 관점으로 보여줄지 정하고, v2.x 대시보드의 모든 기능을 계승하려고 하지 않는다.

### 9. 구현 계획 작성

문서 결정이 충분히 잠긴 뒤 구현 순서를 세운다. 이 단계부터 파일 구조, 테스트 전략, 작업 단위, 커밋 단위를 구체화한다.

### 10. 구현

구현은 문서와 계획을 기준으로 진행한다. 작업 중 새 판단이 필요하면 코드를 먼저 늘리지 않고 관련 문서를 갱신한다.

### 11. Dogfood

Geas 자체를 사용해 실제 작업 하나를 수행한다. 이 단계의 목적은 기능 과시가 아니라 흐름이 실제로 가볍고 이해 가능한지 확인하는 것이다.

### 12. v3.0.0 릴리스 준비

README, changelog, 설치 경로, release workflow, 버전 정보를 정리한다. v3.0.0이 breaking change라면 그 사실과 legacy 위치를 명확히 알린다.

## 4. 진행 원칙

- 한 번에 한 단계씩 진행한다.
- 각 단계의 결정은 문서로 남긴다.
- 문서 없이 구현하지 않는다.
- 구현 중 생긴 새 결정은 문서로 되돌린다.
- 단순한 길이 있으면 그 길을 우선한다.
- v2.x의 이름, 상태, artifact를 필요 없이 되살리지 않는다.
- 부채를 줄이는 것이 새 기능을 늘리는 것보다 우선이다.
